# ⚖️ 本项目的绝对开发宪法 (Highest Constitution)

> 本文件是当前项目的本地开发宪法。每次开始项目开发任务前先阅读，后续实现、验证、汇报都必须服从这里的边界。

## 1. 目标与验收先行

- 先把用户目标改写成一句可验证的工程目标，再开始改代码。
- 每个修改都必须能追溯到用户目标、已确认缺陷或必要质量约束。
- 不添加用户没有要求的功能、架构、依赖、页面、接口或部署改动。
- 不确定需求、数据流、运行态或边界时，先调查；调查仍不确定时，停止并报告最小阻塞点。

## 2. 理解优先于编码

- 修改前必须阅读相关代码、配置、测试、接口契约和现有约定。
- Bug 修复应尽量先复现或定位失败路径，再提出修复。
- 先确认真实运行进程、真实构建产物、真实接口返回或真实 UI 状态，不用“源码看起来对”代替验证。
- PowerShell/终端输出乱码时，优先用 UTF-8 方式重读，不要基于乱码误判需求。

## 3. 工具路由

- CodeGraph：项目结构、调用链、架构、业务流、影响面分析优先使用；修改后可用时刷新/复查索引。
- `rg`：精确文本、变量名、错误日志、配置键、路由名、测试名搜索优先使用。
- gstack/Browser：仅用于 Web UI、浏览器交互、响应式、表单、截图和部署页面验证。
- 原生/设备类任务：按真实目标选择 HTTP、ADB、构建工具、日志或产物校验，不把浏览器验证当作万能替代。

## 4. 边界与审批

- 严禁未经授权跨应用、跨端、跨服务、跨数据库/schema、跨公共 API 契约或跨部署配置修改。
- 读取相关组件用于诊断是允许的；一旦需要修改跨边界文件，必须先给出具体修改计划并等待确认。
- 不覆盖用户已有改动；遇到脏工作区时，先识别哪些变更不是自己做的，并绕开或兼容它们。
- 不执行破坏性命令，不做大范围格式化、重命名、依赖替换或机会主义重构，除非用户明确要求。

## 5. 小步、简单、可回退

- 每次只推进一个主要问题；多个任务只有在完全独立时才并行。
- 优先沿用项目现有框架、目录、命名、helper、测试风格和数据流。
- 只在能减少真实复杂度或重复时引入抽象；不为假想未来搭架子。
- 临时诊断代码、日志、脚本在完成前清理，除非它们有长期运维价值。

## 6. 绝对路径与文件操作

- 汇报和关键操作使用项目根目录起的绝对路径。
- 手工编辑优先用补丁方式；写入前确认目标路径在当前任务范围内。
- 配置、密钥、环境变量、构建产物和发布文件必须区分清楚，不能误改主 LLM/API key 或生产凭据。

## 7. 验证是交付的一部分

- 不能仅凭代码审阅宣称完成；必须报告实际执行过的验证。
- 根据风险选择验证层级：单测、类型检查、构建、接口调用、UI 操作、设备截图、产物哈希、发布地址等。
- 修复类任务优先验证同一失败路径已经恢复。
- 无法验证时，明确说明原因、已完成的静态检查和剩余风险。

## 8. 状态记录与交接

- 每个有代码或配置变更的任务都维护 `PROJECT_DEVELOPMENT.md`。
- 每个有意义增量后更新：当前目标、已验证步骤、下一步计划、风险/Blocker、关键决策。
- 用户要求暂停、保存进度、停止推进时，立即更新状态并停止，不继续额外实现或验证。
## 9. 可复用工作流程沉淀

当某类开发工作出现“反复调整后才成功”的迹象时，必须把它从一次性经验沉淀为可复用工作流程。

### 9.1 触发条件

满足任一条件时，在任务完成并验证后整理流程：

- 同一目标连续经历 3 次或更多“修改 -> 验证 -> 失败/偏差 -> 再调整”后才成功。
- 属于明显会重复出现的常规工作，例如发布、图库模块、上传模块、部署、数据迁移、设备验证、接口联调。
- 本次成功依赖特定命令顺序、运行态判断、目录/端口/设备/产物路径、权限处理或容易遗忘的排错步骤。
- 用户明确要求“记住这个流程”“以后按这个办法”“整理成流程”。

### 9.2 全局工作流程总库

- 总库路径：`C:\Users\Administrator\.codex\workflows\`
- 目录分类：
  - `release/`：版本发布、打包、签名、上传、manifest、远端校验。
  - `module-gallery/`：图库、相册、照片中心、媒体列表、播放/预览链路。
  - `module-upload/`：上传、导入、文件处理、对象存储、批处理。
  - `debugging/`：反复排错、运行态错配、缓存/进程/端口问题。
  - `verification/`：浏览器、HTTP、ADB、截图、产物 hash、部署后 canary。
  - `deployment/`：部署平台、环境变量、CI/CD、回滚、健康检查。
  - `integration/`：跨服务、外部系统、第三方 API、设备或 NAS 集成。

### 9.3 命名与归档格式

- 文件名使用：`YYYY-MM-DD-简短英文slug.md`，例如 `2026-06-15-android-tv-release-artifact-sync.md`。
- 每篇流程必须写清：分类、适用项目/模块、触发场景、技术背景、前置条件、步骤、验证证据、常见失败、回滚/恢复、最后验证日期。
- 流程只能记录已实际验证成功的方法；未验证猜测必须放在“风险/待确认”中。
- 如果流程涉及跨端、数据库、公共 API、生产配置或发布系统，必须注明审批边界和禁止自动执行的步骤。

### 9.4 沉淀时机

- 不要在问题尚未解决时把半成品写成流程。
- 在最终验证通过、临时诊断清理、`PROJECT_DEVELOPMENT.md` 更新后，再写入或更新流程库。
- 若只是本项目局部知识，同时更新项目内 `PROJECT_DEVELOPMENT.md`；若可跨项目复用，再写入全局 `workflows`。

### 9.5 调用方式

开始类似任务时，先用 `rg` 搜索总库：

```powershell
rg -n "发布|release|图库|gallery|上传|upload|设备|artifact|manifest" C:\Users\Administrator\.codex\workflows
```

如果命中相关流程，先阅读对应文档，再制定本轮计划；如果流程已过期或不适配当前项目，必须说明差异而不是机械套用。
