# Android TV 1.0 and Feiniu GHCR Deployment Design

## Goal

Release Android TV version `1.0`, make remote updates resolve through the
deployed Feiniu backend, and provide a production Docker Compose package that
can pull public GHCR images without a GitHub token.

## Approved Decisions

- Android TV uses `versionName "1.0"` and a `versionCode` greater than the
  current value `4`.
- Deployment uses two public images:
  - `ghcr.io/zsdd2/jdyk-backend:1.0`
  - `ghcr.io/zsdd2/jdyk-admin:1.0`
- Both images also publish a moving `latest` tag.
- Feiniu exposes backend port `3999` and admin port `5200`.
- Persistent paths are relative to the Compose directory:
  - `./data/backend`
  - `./data/media-cache`
  - `./data/releases`
- The primary APK download URL is hosted by Feiniu:
  `http://<FEINIU_HOST>:3999/releases/wangri-tv-1.0.apk`.
- GitHub Release `tv-v1.0` remains the backup APK source.

## Architecture

### Backend Image

The backend image contains the compiled NestJS application and production
dependencies. It runs as a non-root user where practical and listens on
`0.0.0.0:3999`.

The container receives persistent mounts for:

- `/app/apps/backend-api/data` from `./data/backend`
- `/app/media-cache` from `./data/media-cache`
- `/app/releases` from `./data/releases`

The backend exposes APK files only from `/app/releases` through the controlled
route `/releases/:fileName`. Path traversal and non-APK access are rejected.

### Admin Image

The admin image contains the production Vue build served by Nginx on container
port `8080`. Feiniu maps host port `5200` to this port.

Nginx proxies `/api/` to `http://backend:3999/api/`. This keeps browser API
requests on the same origin as the admin UI while port `3999` remains available
to Android TV devices.

### Compose Package

The production Compose file contains only `backend` and `admin`. It does not
install Node dependencies or build source on the NAS.

The backend image is parameterized with:

- Feiniu photo-source credentials
- AI provider settings
- Android update metadata
- the public Feiniu host used to form the APK URL

An `.env.feiniu.example` file documents required values. Secrets remain in the
local `.env.feiniu` file and are not committed.

## Android Release and Update Flow

1. Android is built as signed version `1.0`, with a new `versionCode`.
2. GitHub Actions creates `wangri-tv-1.0.apk`, SHA256, file size, and
   `latest.json`.
3. The workflow publishes the APK and manifest to GitHub Release `tv-v1.0`.
4. The Feiniu administrator downloads or copies those files into
   `./data/releases`.
5. The backend update endpoint returns:
   - `versionCode`
   - `versionName`
   - Feiniu-local `apkUrl`
   - `sha256`
   - `sizeBytes`
   - release notes and publication time
6. Android requests:
   `http://<FEINIU_HOST>:3999/api/device/app-update/latest`.
7. Android downloads the APK from the local Feiniu route and validates size and
   SHA256 before opening the system installer.

The GitHub Release URL is documented as a manual fallback. The backend must not
silently redirect to GitHub because local-network-only TVs may not have reliable
GitHub access.

## Image Build and Publication

A GitHub Actions workflow builds both production images on pushes to `main`,
version tags, and manual dispatch:

- `ghcr.io/zsdd2/jdyk-backend:1.0`
- `ghcr.io/zsdd2/jdyk-backend:latest`
- `ghcr.io/zsdd2/jdyk-admin:1.0`
- `ghcr.io/zsdd2/jdyk-admin:latest`

The workflow uses `packages: write`, logs in with `GITHUB_TOKEN`, and publishes
multi-architecture images for `linux/amd64` and `linux/arm64`.

Image definitions are separate because backend and admin have different
runtimes and release lifecycles. Both use multi-stage builds and copy only
required production artifacts into final images.

## Installation and Upgrade

Initial installation on Feiniu:

```sh
cp .env.feiniu.example .env.feiniu
mkdir -p data/backend data/media-cache data/releases
docker compose --env-file .env.feiniu -f docker-compose.feiniu.yml pull
docker compose --env-file .env.feiniu -f docker-compose.feiniu.yml up -d
```

Application upgrade:

```sh
docker compose --env-file .env.feiniu -f docker-compose.feiniu.yml pull
docker compose --env-file .env.feiniu -f docker-compose.feiniu.yml up -d
```

Persistent data is not recreated during image upgrades.

## Failure Handling

- Containers use health checks; admin starts only after the backend is healthy.
- Missing Feiniu or AI credentials do not enter image layers.
- Missing release metadata returns the existing empty update manifest instead
  of advertising a broken update.
- The APK route returns `404` for missing files and rejects unsafe file names.
- Android does not offer an update unless the remote `versionCode` is greater
  than the installed one.
- APK size or SHA256 mismatch prevents installation.

## Verification

### Static and Unit Verification

- Backend update manifest tests pass.
- APK static-route tests cover success, missing file, invalid extension, and
  path traversal.
- Android manifest generation tests read version `1.0`.
- Admin production build and backend production build pass.
- `docker compose config` validates the Feiniu package.

### Image Verification

- Build both images locally.
- Start the Compose stack with temporary persistent directories.
- Verify backend health on port `3999`.
- Verify admin page and `/api/` proxy on port `5200`.
- Verify the APK route returns the expected file and headers.
- Verify image architecture manifests include `amd64` and `arm64` after GHCR
  publication.

### Android Verification

- Build the signed release APK and verify its signature.
- Confirm `versionName=1.0` and the new `versionCode`.
- Generate and compare SHA256 and file size.
- Check the update endpoint over HTTP.
- Install the previous APK on Android TV, trigger update, download from Feiniu,
  and verify the system installer opens.
- Use `F:\Android\Sdk\platform-tools\adb.exe` for device checks and screenshots.

## Scope

This work changes Android versioning, backend release serving, admin production
serving, Docker/GHCR workflows, Feiniu deployment files, and release
documentation. It does not add a graphical release-management page or automatic
copying from GitHub Release into the Feiniu volume.
