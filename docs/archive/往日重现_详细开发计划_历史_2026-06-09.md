# 《往日重现》详细开发计划

更新时间：2026-06-09

## 1. 当前确定路线

项目采用“调度中枢 + Web 管控端 + Android TV 放映端 + 私有照片源”的路线。产品名正式确定为“往日重现 (Yesterday Once More)”，定位从单纯照片播放工具升级为家庭影像的数字策展与电视展播系统。

- 调度中枢：2.0 目标形态确定为 Python FastAPI + PostgreSQL + Redis/Celery/APScheduler；当前仍以 NestJS `apps/backend-api` 作为业务验证桥接层，继续承载照片源、设备、播放策略、AI 任务和媒体代理，后续按 `kf01/api-contracts/openapi.yaml` 迁移，不推翻已验证契约。
- Web 管控端：基于 Vben Admin 的 `apps/web-antd` 改造为“策展人控制台”，负责混合源资产、照片中心、AI 评语/标题、播放相册、TV 展示排障和后续设备授权。
- Android TV 放映端：`apps/android-tv` 作为真实电视端安装包和长期主播放器，当前阶段继续使用“后端地址 + 账号密码”登录经典一刻后台，负责拉取播放内容、展示 AI 评语/标题和沉浸放映；不再把 PIN 绑定作为近期主线。
- 私有照片源：本地/NAS 扫描照片、飞牛 fnOS 相册、共享相册和后续其他来源统一进入照片中心索引。
- Web 播放页：当前 `apps/tv-player-web` 调整为后台播放预览、浏览器调试和备用 WebView 模式，不再承担正式放映端职责。
- 共享协议：`packages/shared` 维护播放单元、设备策略、播放记录等跨端类型。
- 照片源层：新增 `FeiniuSource` / 飞牛相册源，复用 `fnphoto-tv-main` 中已跑通的飞牛登录、`authx` 签名、相册接口、共享相册接口和媒体 URL 规则。
- 来源抽象：后端把本地 SQLite/扫描照片和飞牛相册统一适配成 `AlbumSummary`、`AlbumDetailResponse`、`PlaylistItem`，Android TV 继续只消费经典一刻自己的 `/api/device/albums`、`/api/device/albums/:albumId` 和 `/api/device/playlist` 契约。

核心判断：当前工程不推倒重来，而是按 2.0 白皮书做兼容迁移。已完成的照片中心、飞牛源、共享相册、播放相册、快速分拣台、AI 设置中心、相册 AI 开关和 TV 账号密码登录都保留；未完成的能力改按“多源入池 -> 三规格派生图 -> 动态映射 -> 限量 AI 提纯 -> 广域网鉴权 -> 质量优先下发 -> TV 边缘放映”的主业务线重排。`fnphoto-tv-main` 继续作为飞牛协议参考工程，不迁移它的 Leanback 展示层、播放页和 UI 信息架构。

Android TV 详细技术选型见 `往日重现_AndroidTV技术选型.md`。

## 1.0 2.0 最高指导纲领

2.0 的项目最高指导目录已建立在 `kf01/`：

- `kf01/README.md`：统一产品定位、业务主线、模块边界和近期落地顺序。
- `kf01/DEVELOPMENT_CONSTITUTION.md`：规定契约先行、数据先行、媒体瘦身、AI 漏斗、兼容策略和验证要求。
- `kf01/api-contracts/openapi.yaml`：先行定义 Admin、TV Auth、播放相册、AI 设置、AI 任务和每日主题流的契约草案。

从现在开始，跨端接口新增或字段修改必须先同步 `kf01/api-contracts/openapi.yaml`，再写后端测试和实现。当前 NestJS + SQLite 只作为 2.0 行为桥接层，后续迁移到 FastAPI + PostgreSQL 时优先保持契约兼容。

### 1.0.1 2.0 主业务流水线

```text
物理源：飞牛 NAS / 本地目录
-> 多源入池：飞牛相册清单同步，本地目录扫描，飞牛照片按需导入
-> 异步三规格派生图：300px 管控端缩略图 + 720p AI 分析图 + 4K WebP TV 放映图
-> 动态映射：虚拟播放相册可以包含本地分拣照片，也可以直接挂载飞牛相册
-> 限量提纯：每日最多 100 张未处理照片进入 AI 漏斗
-> 广域网鉴权：TV 输入后端地址、账号、密码，获取长效设备 JWT
-> 质量优先下发：每日最多 10 张，不足时用历史 90+ 高分经典图补齐
-> 边缘放映：TV 本地缓存 daily_themes.json 和 4K WebP，次日 00:00 清理
```

### 1.0.2 已完成能力的迁移判断

- 照片中心：保留为资产事实层；后续从 SQLite 表迁移到 PostgreSQL，字段扩展为指纹、EXIF、衍生图路径、AI 漏斗状态和淘汰状态。
- 照片扫描：保留为单独页面和本地目录入池动作，不与照片中心列表合并。
- 飞牛相册源：保留 `FeiniuSource`、登录、authx、共享相册和媒体代理；飞牛扫描继续只拿相册，不全量读取照片。
- 播放相册：保留手动分拣和飞牛相册挂载两种模式；飞牛照片在查看成员、生成 AI、推送每日内容时按需导入。
- 快速分拣台：保留当前穿梭式分拣工作台，后续接入 AI 类型标签、评分线、时间、主题和来源筛选。
- AI 设置中心：保留平台、模型、接口地址、API Key、评分提示词、点评提示词、类型识别提示词；统一 Vision AI Adapter 已完成首版，优先用一次请求完成评分、点评、分类和 TV 设计，效果不足时再拆回多阶段提示词。
- Android TV：保留“后端地址 + 账号密码”登录；PIN 绑定不作为近期主线。后续登录结果升级为 JWT，并增加 daily_themes 静态流同步。

### 1.0.3 未完成能力的目标修订

1. 衍生资产层：新增照片指纹、EXIF、300px 缩略图、720p AI 分析图、4K WebP TV 图路径和转码状态。
2. AI 漏斗层：优先实现“单次 Vision JSON -> 评分/点评/分类/布局 -> 过线判断 -> 持久化”的省流量通路；如果真实照片测试效果不稳定，再退回评分与排版分离的两阶段调用。
3. 调度层：新增每日 01:00 AI 限量任务、04:00 每日主题生成、历史高分兜底。
4. 下发层：新增 `/api/device/daily-themes`，返回 TV 专属每日 5-10 张静态流。
5. TV 展示层：读取布局位置、字体风格和文本颜色，做避开人物主体的安全区展示；本地缓存跨日清理。
6. 架构迁移层：在契约和测试稳定后，再拆出 FastAPI + PostgreSQL + Redis/Celery/APScheduler。

## 1.1 优化后的开发模式

从 2026-06-05 起，项目开发模式调整为“主业务闭环优先”。

核心原则：先只做能证明产品成立的最短业务流，允许临时内存数据、样例图片和调试接口；等 Docker 后台和 Android TV APK 能完整跑通后，再逐步替换为真实数据库、NAS 扫描、管理后台和 AI 功能。

### 1.1.1 最短业务闭环

本次路线调整后，第一优先级改为“AI 评语 + TV 展示闭环”，只服务这条链路：

```text
混合照片源导入
-> 照片中心索引
-> 选择一批真实照片或播放相册
-> 生成/保存 AI 标题、AI 评语、评分和展示设计数据
-> TV 通过后端地址 + 账号密码登录经典一刻后台
-> TV 拉取带 AI 文案和展示参数的播放流
-> Android TV 用接近最终产品的视觉效果沉浸放映
```

只要这条链路没有跑通，以下能力都不作为主线任务：

- 泛化的照片社交/分享能力。
- 与每日十图无关的复杂播放器模板。
- PIN 绑定和多设备授权白名单。
- 复杂播放相册成员后台管理。
- 完整商业 SaaS 租户系统。
- 大规模 NAS 增量扫描优化。
- 高级动画模板。
- 自动发现后台地址。
- 完整云端账号体系和付费系统。
- Docker 镜像体积优化和长期运维能力。

### 1.1.2 每轮开发规则

- 每轮只推进一个可验证业务节点，例如“TV 拉取播放列表并显示第一张图”。
- 可以使用临时实现，但必须在 `往日重现_项目开发进度.md` 记录限制和未来替换点。
- 涉及后端、Android TV、共享协议时，可以在同一轮联动修改，但必须保持目标是同一条业务链路。
- 每轮完成后至少做构建验证；涉及接口时做接口实测；涉及 Android TV 时做 APK 构建。
- 每轮结束都刷新 CodeGraph，并更新开发进度文档。

### 1.1.3 新的阶段压缩

原 8 个阶段保留为长期路线，但实际开发按下面 6 个大阶段推进：

1. 混合源和照片中心：本地/飞牛照片统一进入资产池，具备来源、相册、时间、AI 状态和媒体 URL。
2. AI 评语和展示数据：先用手动/占位 Provider 给照片生成标题、评语、评分和展示布局字段。
3. TV 展示闭环：TV 用后端地址和账号登录，拉取带 AI 文案的播放流，并完成沉浸展示效果验证。
4. 策展和播放相册：照片中心支持筛选、批量选择、加入播放相册、人工锁定和基础维护。
5. AI 优先级调度和每日内容：相册优先级、插队处理、暂停/恢复、任务额度、每日十图和后续设备差异化。
6. 生产化和技术栈升级：PostgreSQL/Redis/APScheduler 或 FastAPI 迁移评估、Docker 部署、缓存、备份、长时间播放。

当前最优先任务：`PhotoSource` 抽象、`FeiniuSource` 后端 MVP、共享给我的飞牛相册接入、经典一刻后端媒体代理、mixed 聚合照片源、照片中心列表 API、后台三页拆分、播放相册后端模型/API、照片中心批量加入播放相册首版交互、播放相册管理页、快速分拣台、飞牛相册挂载、相册级 AI/推送策略首版、AI 展示数据契约首版、本地成员占位 AI 生成 MVP、播放相册成员移除首版、飞牛挂载成员合并展示、AI 开关自动生成、AI 设置中心、飞牛挂载照片按需导入并持久化 AI 状态、统一 Vision AI 单次调用契约和三规格派生图字段已完成。下一步暂停 PIN 绑定、复杂设备授权和播放相册深度管理，优先做“真实三规格图片转码 -> 真实 Vision API 小样本测试 -> 播放流字段级 HTTP 验证 -> Android TV 展示效果”这一条 2.0 主业务闭环。

## 1.2 飞牛相册源重构路线

### 1.2.1 重构目标

以 `jdyk` 作为主项目，在后端新增 `FeiniuSource`，把飞牛 NAS 相册作为一种真实照片源接入。TV 端继续使用经典一刻自己的登录页、图包选择页、图包详情页、全屏播放页、播放设置页和播放记录上报流程，不直接调用飞牛 NAS。

### 1.2.2 复用边界

- 复用 `fnphoto-tv-main/app/src/main/java/com/fnphoto/tv/api/FnWebSocketClient.java` 的登录流程：连接飞牛 WebSocket，获取 RSA 公钥，AES/RSA 加密账号密码，拿到 `token`、`secret` 和会话信息。
- 复用 `fnphoto-tv-main/app/src/main/java/com/fnphoto/tv/api/FnAuthUtils.java` 的 `authx` 规则：6 位 nonce、毫秒时间戳、GET 参数按 key 排序、payload MD5、`API_KEY_path_nonce_timestamp_payloadHash_API_SECRET` 再 MD5。
- 复用 `fnphoto-tv-main/app/src/main/java/com/fnphoto/tv/api/FnHttpApi.java` 的相册接口：`/p/api/v1/album/list`、`/p/api/v1/album/photos`、共享相册列表，后续可扩展时间线、文件夹和搜索。
- 复用媒体 URL 规则：优先使用飞牛返回的 `additional.thumbnail.originalUrl` / `mUrl` / `sUrl` / `xsUrl`，相对路径统一补齐为飞牛 NAS base URL。
- 不复用 `fnphoto-tv-main` 的 Android TV UI、Leanback 导航、Glide/ExoPlayer 播放流程和本地缓存策略，避免破坏经典一刻已有 TV 产品体验。

### 1.2.3 目标架构

```text
Android TV APK
  -> 经典一刻后端 /api/device/albums
  -> AppService / PhotoSourceRegistry
  -> FeiniuSource
  -> 飞牛 WebSocket 登录 + authx HTTP API
  -> 飞牛相册/照片/媒体 URL

Android TV APK
  -> 经典一刻后端 /api/device/playlist 或 /api/device/albums/:albumId
  -> 统一 PlaylistItem
  -> 经典一刻全屏播放/展示/播放记录上报
```

### 1.2.4 计划触达文件

- `packages/shared/src/index.ts`：补充照片源类型、飞牛源配置、来源标识和错误状态类型。
- `apps/backend-api/src/photo-sources/photo-source.ts`：新增统一照片源接口，定义 `listAlbums`、`getAlbum`、`listPlaylistItems`、`getMediaAsset` 或媒体 URL 解析边界。
- `apps/backend-api/src/photo-sources/sqlite-photo-source.ts`：把现有 `SqlitePhotoRepository` 包装为默认本地照片源，保持当前 `ceshi` 和扫描数据行为不变。
- `apps/backend-api/src/photo-sources/feiniu/feiniu-auth.ts`：移植 WebSocket 登录和 `authx` 签名逻辑到 Node/NestJS。
- `apps/backend-api/src/photo-sources/feiniu/feiniu-client.ts`：封装飞牛 HTTP API、token 缓存、401 重新登录和分页。
- `apps/backend-api/src/photo-sources/feiniu/feiniu-source.ts`：把飞牛相册和照片转换成 `AlbumSummary`、`AlbumDetailResponse`、`PlaylistItem`。
- `apps/backend-api/src/app.service.ts`：从直接依赖 `SqlitePhotoRepository` 改为依赖照片源注册表，默认仍使用本地源，启用飞牛配置后切到 `FeiniuSource`。
- `apps/backend-api/src/app.controller.ts`：保持 TV 端接口路径和响应结构不变，只在需要时增加后台管理端的照片源配置/连通性检测接口。
- `apps/backend-api/src/*.spec.ts`：增加 `authx` 签名、飞牛响应映射、token 失效重登、TV 契约不变的单元测试。

### 1.2.5 分阶段实施

1. 照片源接口先行：抽象 `PhotoSource`，把现有 SQLite 数据源包装进去，确保现有后端测试和 Android TV 契约完全不变。
2. 飞牛认证最小闭环：实现 Node 版 WebSocket 登录、AES/RSA 加密、`authx` 签名，并用单元测试锁定签名排序和 payload hash 规则。
3. 飞牛相册列表接入：实现 `/p/api/v1/album/list` 到 `AlbumSummary` 的映射，先让 TV 图包选择页看到飞牛相册。
4. 飞牛相册详情接入：实现 `/p/api/v1/album/photos` 到 `PlaylistItem` 的映射，生成 `displayImageUrl`、`imageUrl`、`thumbnailUrl`。
5. 媒体 URL 代理：飞牛媒体直链需要 `accesstoken` / `authx`，TV 端不能直接生成飞牛鉴权头；当前已切换为经典一刻后端媒体代理，统一给 Coil 返回可直接访问的 `/api/photos/:photoId/display|thumb|original`。
6. 配置和管理入口：后台增加飞牛 NAS 地址、账号、密码/凭据保存、连通性检测和当前照片源选择；敏感信息不能写入前端日志。
7. TV 端回归验证：保留 TV 现有页面，只验证登录经典一刻后端、图包选择、详情、播放、切换、空状态和播放记录。

### 1.2.6 验收标准

- 未配置飞牛时，现有 SQLite/`ceshi` 行为、测试数据、空状态 demo 和 Android TV 流程保持不变。
- 配置飞牛后，`GET /api/device/albums` 返回飞牛相册，字段仍符合 `AlbumListResponse`。
- `GET /api/device/albums/:albumId` 返回飞牛相册内照片，字段仍符合 `AlbumDetailResponse` 和 `PlaylistItem`。
- Android TV 不需要知道飞牛 `authx`、飞牛 token、飞牛相册接口路径，也不直接保存飞牛账号密码。
- 飞牛 token 失效时后端能重新登录或返回明确错误，不导致 TV App 崩溃。
- 至少完成后端 Jest 测试、飞牛配置关闭场景回归、飞牛映射 mock 测试；真实飞牛空态 HTTP、共享相册非空 HTTP、媒体代理和 Android TV ADB 截图验证已完成，后续继续补普通相册、我共享的相册、大相册分页和真实 TV 设备长时间播放验证。

## 1.3 照片中心优先路线

### 1.3.1 调整原因

当前后台只有“照片扫描”会让管理端能力过窄：用户真正需要的是统一看见本地照片和飞牛照片，知道哪些照片已经 AI 打分、哪些已经有 AI 评语，并能按导入相册、时间、主题、评分等维度快速分拣到具体相册。因此下一阶段管理后台从“扫描入口”升级为“照片中心”。

### 1.3.2 第一阶段目标

- 同一照片中心列表中展示本地扫描照片和飞牛照片。
- 每张照片有明确来源：本地、飞牛普通相册、共享给我的相册、我共享的相册。
- 每张照片记录 AI 状态：未评分、已评分、评分失败、未生成评语、已生成评语、人工锁定。
- 支持筛选：来源、导入相册、拍摄时间、主题、评分区间、有无评语、是否已分拣。
- 支持批量动作：加入已有相册、创建新相册、标记忽略、重新评分、重新生成评语。

### 1.3.3 AI 打分和评语机制

AI 打分和评语不在 TV 播放时实时生成。推荐机制是“定时检测 + 手动触发”：

```text
照片扫描 / 飞牛同步
-> 写入照片中心索引
-> 发现未评分或未评语照片
-> 后台任务队列批量处理
-> 保存评分、主题、标题、评语和状态
-> TV 播放时只读取已保存结果
```

这样可以控制 AI 成本、避免 TV 端等待、支持失败重试和人工锁定。后续可增加每日额度、并发限制、失败重试和任务日志。

### 1.3.4 分阶段实施

1. 先扩展 SQLite schema：照片来源、来源相册 ID、AI 评分、主题标签、评语状态、人工锁定字段。

## 1.4 白皮书目标架构

### 1.4.1 产品定位

“往日重现 (Yesterday Once More)”的最终形态是家庭影像展播系统，不是普通相册浏览器。核心价值是把沉睡在 NAS、飞牛相册和本地硬盘里的照片，通过 AI 评审、人工策展和电视授权机制，变成每天可在家庭中心屏幕上播放的高质量回忆流。

### 1.4.2 目标模块

- 调度中枢：负责照片中心索引、AI 评语/评分、播放流生成、媒体代理、缓存，以及后续 AI 任务队列、设备授权和每日十图。
- Vben 管控端：负责混合源资产管理、照片中心、AI 评语结果校验、播放相册、TV 展示排障、系统设置和后续设备授权。
- Android TV 放映端：负责后台地址配置、账号密码登录、播放相册/播放流拉取、AI 文案展示、每日内容缓存和沉浸放映。
- 私有照片源：本地扫描、NAS 挂载、飞牛普通相册、飞牛共享相册以及后续可扩展照片源。

### 1.4.3 技术栈调整原则

白皮书提出的 Python FastAPI + PostgreSQL + APScheduler 是目标架构候选，但当前阶段不直接重写已跑通的 NestJS 后端。原因是飞牛协议、TV 图包契约、媒体代理、照片中心索引和 Android TV 链路已经在当前后端中验证通过。后续按以下闸口推进：

1. 当前阶段继续用 NestJS + SQLite 做业务闭环，优先验证产品形态。
2. 当 AI 评语、TV 展示闭环、每日内容生成三条主链路稳定后，评估 PostgreSQL 作为正式数据层。
3. 如果 AI 调度、定时任务、批量处理复杂度明显超过 NestJS 当前结构，再拆出 Python FastAPI/Worker 调度中枢。
4. 迁移时保持 TV 和 Vben 对外 API 契约稳定，不要求 Android TV 因后端技术栈变化重写。

## 1.5 TV 登录、AI 展示与每日内容

### 1.5.1 目标流程

```text
TV 首次打开或配置页打开
-> 用户输入经典一刻后端地址、账号、密码
-> TV 调用 `/api/device/login` 获取设备 Token
-> TV 拉取播放相册或 AI 展示播放流
-> 播放项包含照片 URL、标题、AI 评语、评分、拍摄时间和展示参数
-> TV 用最终产品方向的版式和动效沉浸播放
-> 后续再把每日内容缓存、设备差异化和授权白名单接入同一播放流
```

### 1.5.2 数据边界

- 登录：TV 保存经典一刻后端地址、账号和设备 Token；不保存飞牛账号、飞牛 token 或 `authx`。
- 展示播放项：必须包含 Photo ID、展示图 URL、标题、AI 评语、评分、时间、来源和版式/文案显示参数。
- 当前播放范围：第一版可以直接使用指定播放相册或精选照片池，不要求多设备差异化。
- 后续授权：设备白名单和订阅保留为生产化扩展，不再用 PIN 作为近期主线入口。
- 每日内容：每日十图后续按播放范围、AI 分数、人工锁定、播放历史和多样性规则生成。
- 缓存：TV 后续每天拉取一次 daily playlist；图片缓存可本地过期，次日 00:00 后清理旧数据。

### 1.5.3 接口方向

- `POST /api/device/login`：TV 用经典一刻后台账号登录并获取设备 Token。
- `GET /api/device/albums` / `GET /api/device/albums/:albumId`：短期继续作为 TV 选择和播放入口。
- `GET /api/device/playlist?albumId=...`：短期扩展为返回 AI 标题、AI 评语、评分和展示参数。
- `GET /api/device/daily-playlist`：后续在展示闭环稳定后再接入每日内容。
- `POST /api/admin/devices/:deviceId/authorized-albums`、`GET /api/device/authorized-albums`、`POST /api/device/subscriptions`：保留为后续多设备授权能力，不进入近期主线。

当前接口命名不必一次性改成白皮书里的 `/api/auth/tv/bind`。近期原则更简单：TV 先稳定连接经典一刻后端并播放带 AI 文案的内容；等展示效果和内容质量成立后，再补设备授权和每日十图。

## 1.6 AI 优先级调度中心

### 1.6.1 目标能力

- 相册权重：High / Normal / Low，决定 AI 任务投喂顺序。
- 插队处理：后台可对新导入相册点击“优先处理”，优先消耗当日额度。
- 暂停/恢复：废片多的相册可暂停 AI 评估，节省 Token。
- 进度看板：显示每个相册已处理/未处理、失败数、平均分、高分库存。
- 任务日志：记录模型、成本、失败原因、重试次数和人工锁定状态。

### 1.6.2 调度规则

第一版可用 SQL/任务表表达：

```sql
ORDER BY priority DESC, created_at ASC
LIMIT 100
```

正式规则需要叠加：

- 只处理未评分或未生成评语的照片。
- 跳过人工锁定、忽略、暂停相册。
- 高优先级相册未处理完前，优先占用当日 AI 名额。
- 每日十图只从已评分、未忽略、达到最低分数线的照片中选择。

### 1.6.3 Vben 菜单目标

- 目标概览：昨日 AI 评阅量、高分库存、可播放天数、Token 消耗。
- 混合源资产管理：本地源、飞牛源、同步状态和错误。
- 照片中心：全量资产、筛选、批量分拣、AI 状态。
- AI 调度中心：任务队列、优先级、暂停/恢复、进度看板。
- 设备与授权：设备列表、后端账号登录产生的设备身份、相册授权、订阅状态；PIN 不作为近期主线。
- 往日画廊：高分照片、AI 评语、人工修订和版式预览。

- 扫描日志、失败文件、重扫机制。

### 开发顺序

1. 先实现本地挂载目录扫描。
2. 再实现飞牛源同步和照片中心持久化索引。
3. 再实现缩略图生成。
4. 再实现 EXIF。
5. 最后做增量扫描、失败重试和来源去重。

### 验收标准

- 后台能添加一个本地挂载照片目录。
- 扫描/同步后照片中心出现真实照片。
- 缩略图能正常显示。
- 播放列表能播放真实照片。
- 读取失败、格式不支持、EXIF 缺失都有日志。

### 主要风险

- NAS 目录可能非常大，扫描必须分批和可中断。
- 原图不能删除或移动，只能建立索引和缓存。
- 缩略图生成要控制内存和并发。

## 8. 阶段 5：AI 评语、展示设计和 TV 主体验

### 阶段目标

让照片不只是“能播放”，而是能带着可信的 AI 标题、AI 评语、评分和展示设计在 Android TV 上形成完整体验。后台只做支撑主体验所必需的入口，不优先扩展复杂管理。

### 核心任务

- AI 展示结果字段：标题、评语、评分、主题标签、文案状态、人工锁定、展示模板。
- AI Provider MVP：先支持手动触发一批照片生成结构化结果；如果真实模型暂不可用，可先做本地占位 Provider 打通字段和 TV 展示。
- 播放流扩展：`PlaylistItem` 或设备播放接口返回 AI 标题、评语、评分和展示参数。
- Android TV 展示页：把照片、标题、评语、日期、来源、评分和动效组合成接近最终产品的沉浸页面。
- TV 播放效果验证：真实飞牛共享相册和本地照片都能展示 AI 文案；遥控器切换和自动轮播不破坏版式。
- 管理端最小入口：照片中心能看到 AI 状态，支持手动生成/重试/锁定；播放相册深度成员管理暂后置。

### 开发顺序

1. 定义 AI 结果和 TV 展示字段，补后端测试。
2. 做 AI 结果生成 MVP：先支持选中照片/相册手动生成标题、评语和评分。
3. 扩展 TV 播放流，让 Android TV 能拿到 AI 文案和展示参数。
4. 改 Android TV 播放页，完成最终方向的版式、层级、渐显和长文截断。
5. 做一轮真实数据验证：本地照片、飞牛共享相册、缩略图/原图代理、自动轮播和遥控器切换。
6. 只有 TV 主体验成立后，再继续补播放相册成员管理、设备授权、每日十图和 AI 调度中心。

### 验收标准

- 一批真实照片可生成或写入 AI 标题、评语和评分。
- AI 结果绑定 Photo ID，TV 播放时不实时调用云端 AI。
- TV 播放页能展示照片、标题、评语、时间、来源和评分，且 1080p 下文字不溢出、不遮挡主体。
- 本地照片和飞牛共享相册都能进入同一展示链路。
- 自动轮播和遥控器切换都能保持展示状态正确。
- 后台能查看哪些照片已有 AI 评语、哪些仍待处理。

### 主要风险

- AI 文案不能编造家庭身份、亲属关系或敏感事实；第一版要克制、短句、可人工修改。
- TV 展示页是主产品体验，不能只做后台字段；必须用真实照片验证视觉层级和可读性。
- AI Provider 调用成本和失败要有上限；第一版可用手动触发和小批量验证。

## 9. 阶段 6：策展管理后台、播放相册和后续设备授权

### 阶段目标

在 TV 主体验成立后，把 Vben Admin 从通用模板继续改造成“策展人控制台”，让用户能把照片中心资产分拣成播放相册，并在后续对不同电视做差异化授权。

### 核心任务

- 精简 Vben 示例模块。
- 目标概览：照片数量、AI 高分库存、可播放天数、最近错误。
- 混合源资产管理：本地源、飞牛源、同步状态、源错误和连通性检测。
- 照片中心：缩略图、来源、AI 状态、筛选、批量选择、批量加入播放相册。
- 播放相册：创建、命名、排序、封面、成员管理、启停。
- 设备与授权：后续使用后端地址/账号登录产生的设备身份，不再把 PIN 绑定作为默认方案；可补设备命名、在线状态、可见相册白名单、订阅状态。
- 播放策略：每日内容开关、播放模式、速度、文案显示、动画模板。
- 日志状态：扫描日志、AI 日志、播放错误。
- 系统设置：后台地址、媒体目录、AI Key、调用额度。

### 开发顺序

1. 强化照片中心批量分拣：筛选持久化、加入历史、快速查看目标播放相册。
2. 补播放相册成员管理：成员移除、排序、封面维护和启停。
3. 基于 TV 登录产生的设备身份做设备列表和在线状态。
4. 再补可见相册白名单和订阅状态，不使用 PIN 绑定作为主入口。
5. 做播放策略、日志和系统设置。

### 验收标准

- 管理端能把照片中心筛选结果加入播放相册。
- 管理端能维护播放相册成员、封面和启停状态。
- TV 登录后可被后台识别为一个设备。
- 后续授权启用后，TV 端只能看到被授权的相册。
- 常见错误能在日志页面查到。
- 页面不保留明显无关的 Vben 示例入口。

### 主要风险

- Vben 模板功能多，精简要控制范围，避免误删核心权限、菜单和请求封装。
- 管理端不是主体验，不能把过多时间耗在视觉细节上。

## 10. 阶段 7：AI 优先级调度、每日内容和离线缓存

### 阶段目标

让每台 TV 都能根据自己的授权和订阅，获取专属的“每日十图”静态播放流，并在本地缓存当天 JSON 和展示图。

### 核心任务

- 默认每日内容表：先保存单 TV / 默认家庭屏幕的每日播放结果，不要求多设备授权差异。
- 每日十图生成任务：先按播放相册、AI 分数、人工锁定和多样性规则生成；后续再叠加设备和订阅。
- 播放历史去重：避免连续多天重复同一组照片。
- Daily playlist API：TV 拉取当天专属播放流。
- TV 本地缓存：保存 `daily_10_photos.json` 和 10 张展示图/WebP。
- 过期清理：次日 00:00 后清理旧 daily 缓存。
- 异常兜底：后台不可达时继续播放本地缓存；缓存缺失时显示明确空状态。

### 验收标准

- 同一台 TV 每天能拉取固定的 10 张照片。
- 后续不同 TV 可因授权/订阅不同得到不同每日十图，但不作为第一版验收条件。
- TV 离线时能播放最近一次成功缓存。
- 每日十图不包含未完成 AI 评审或被忽略照片；授权过滤后续接入。

### 主要风险

- 每日十图需要兼顾高分、时间跨度、相册多样性和避免重复。
- TV 本地缓存要控制容量和清理时机，避免长期占用存储。

## 11. 阶段 8：稳定性、技术栈升级、打包发布和长期运维

### 阶段目标

把项目从可用 MVP 提升到可长期部署在家庭 NAS 和 Android TV 上运行，并在业务闭环稳定后评估 PostgreSQL / Redis / FastAPI Worker 的正式化迁移。

### 核心任务

- PostgreSQL 数据层迁移评估和迁移脚本。
- Redis/任务队列或 APScheduler 调度评估。
- 如需要，拆分 Python FastAPI/Worker 调度中枢，保留现有 TV/Vben API 契约。
- Docker 生产镜像优化。
- 数据备份和恢复说明。
- 后台日志归档。
- Android TV APK 打包签名。
- Android TV 版本号和升级说明。
- 播放端离线缓存播放列表。
- 低端电视性能降级：禁用复杂动画、降低预加载数量。
- 内存释放和长时间播放稳定性测试。
- 网络异常、后台重启、NAS 断开、照片丢失等异常场景测试。
- 项目安装说明和常见问题文档。

### 验收标准

- Docker 后台可按说明部署。
- Android TV APK 可安装并连接后台。
- 连续播放数小时无明显异常。
- 后台重启后电视端能恢复播放。
- 数据库迁移不破坏照片中心、授权、订阅和每日十图数据。
- 常见部署问题有排查说明。

### 主要风险

- 家庭 NAS 环境差异大，路径、权限、网络都可能出问题。
- Android TV 设备性能差异大，需要有降级策略。

5. 已完成首版：本地成员占位 AI 评语/评分生成 MVP，后台 AI 开关打开后自动生成，SQLite/TV 播放流会读取 AI 文案。
