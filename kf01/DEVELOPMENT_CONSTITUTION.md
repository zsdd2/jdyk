# 往日重现 Development Constitution

更新时间：2026-06-09

## 最高开发原则

所有新增代码必须服务 2.0 主业务线：多源入池、三规格派生图、虚拟播放相册、AI 限量提纯、广域网鉴权、质量优先下发、TV 边缘放映。后台管理能力必须围绕这条链路建设，不能把扫描页、配置页或表格管理当作产品主线。

## 契约先行

- 新增或修改跨端接口前，先更新 `kf01/api-contracts/openapi.yaml`。
- 后端实现可以先在 NestJS + SQLite 中落地，但响应结构必须按 2.0 契约设计，便于后续迁移到 FastAPI + PostgreSQL。
- Android TV 只消费往日重现后端契约，不直接依赖飞牛 API、飞牛 token 或 authx。

## 数据先行

- 照片中心是资产事实表；照片扫描只是入库方式，不能和照片中心混成一个页面或一个概念。
- 飞牛相册清单同步不等于飞牛照片全量同步；飞牛照片默认按需导入。
- 所有 AI 结果必须持久化：评分、废片判断、类型标签、点评文案、布局位置、字体风格、文本颜色、处理状态。
- TV 推送和播放只读取已持久化结果，不能在 TV 播放时实时等待 AI。

## 媒体瘦身

- 管控端列表只使用最大长边 300px 的缩略图。
- AI 评分、点评、类型识别和 TV 设计默认使用 720p 分析图，避免把原始大图或 4K 图直接投喂模型。
- TV 放映默认使用最大长边 3840px 的 4K WebP 衍生图。
- 原始大图只作为源文件保留，不直接下发 TV。

## AI 漏斗

1. 当前优先采用统一 Vision 单次调用，输出 `memory_score`、`beauty_score`、`is_trash`、`reason`、类型标签、`copywriting` 和避开人物主体的 `layout`。
2. 低于相册评分线或 `is_trash=true` 的照片进入淘汰状态。
3. 如果真实照片测试证明单次调用的评分或排版不稳定，再拆成“评分初筛”和“文案排版”两阶段调用。
4. 相册可以独立配置 AI 开关、评分线、类型优先级和推送分数线。

## 兼容策略

- 已有照片分拣、播放相册、本地扫描、飞牛共享相册和 TV 登录流程不删除。
- 新功能优先通过兼容字段、适配层或迁移脚本叠加。
- 如果目标架构要求替换底层技术栈，先保证 API 契约和测试覆盖，再迁移存储和运行时。

## 验证要求

- 后端跨端接口必须有 Jest 或后续 FastAPI 测试。
- 前端用户可见改动必须通过类型检查和关键交互测试。
- TV 展示改动必须用 APK 构建验证；涉及真实设备时优先 HTTP + ADB 截图，不用浏览器验证替代。
- 每轮修改后更新 `往日重现_项目开发进度.md` 和 `往日重现_详细开发计划.md`。
