# 《往日重现》开发基础文档

版本：v0.5  
用途：用于 Codex / AI 编程工具快速理解项目目标、业务边界、系统架构、模块拆解与开发计划。  
项目类型：Docker 后台端 + Web 管理端 + Android TV APK 电视端。  
核心定位：NAS 家庭照片回忆播放器，不是完整相册管理系统。
计划调整日期：2026-06-05。

---

## 1. 项目总目标

《往日重现》是一个面向电视大屏的家庭照片回忆播放器。系统通过 Docker 后台连接 NAS 照片目录，扫描照片、生成缩略图、读取 EXIF、调用云端 AI API 识图并生成回忆文案，然后将“照片 + 文案 + 版式 + 动画 + 播放策略”组成播放单元，推送到不同电视设备播放。

核心目标：

1. 让 NAS 中沉睡的家庭照片重新在电视上被看见。
2. 支持多设备播放不同照片目录。
3. 支持照片随机播放、顺序播放、循环播放、手动切换。
4. 支持后台设置播放速度、播放模式、转场动画、文案版式。
5. 支持前端电视端局部覆盖播放速度和部分展示设置。
6. 支持云端 AI 识图，为每张照片绑定标题、一句话、场景标签、情绪标签、推荐版式和推荐动画。
7. 支持后台预览电视端实际播放效果。
8. 保持电视端极简、沉浸、稳定、低干扰。

## 2. 产品边界

### 2.1 本项目要做

- NAS / 本地挂载照片目录读取。
- 照片扫描、索引、缩略图生成。
- EXIF 读取，包括拍摄时间、方向、GPS、尺寸。
- AI 识图资料生成。
- AI 文案与 Photo ID 绑定。
- 多设备管理。
- 后台配置播放目录和播放策略。
- 电视端全屏播放。
- 图片转场动画。
- 文字动画。
- 播放预览。
- 运行日志与错误排查。

### 2.2 第一版不做

- 不做完整云相册。
- 不做社交分享。
- 不做复杂用户权限。
- 不做手机 App。
- 不做视频剪辑。
- 不做在线相册托管。
- 不做云同步。
- 不做复杂人脸识别管理。

### 2.3 产品原则

后台负责管理，电视负责展示。  
照片是主角，文案是辅助。  
AI 资料必须绑定照片，而不是播放时临时生成。  
电视端所有界面必须适配遥控器操作。

### 3.2 推荐技术栈

#### 后台基础框架候选

原则：可以使用若依等成熟后台脚手架，但不预设只能用若依。选择标准是后台起步快、权限和菜单成熟、业务代码不被低代码平台锁死、后台任务和文件处理容易扩展、后续电视端播放链路好打通。

候选方向：

- RuoYi-Vue / RuoYi-Vue3-TypeScript：全栈后台成熟，权限、菜单、日志、代码生成、定时任务现成，适合快速搭后台底座。
- RuoYi-Vue-Plus：若依增强路线，技术栈更现代，内置多租户、OSS、工作流等能力，但第一版可能偏重。
- Vben Admin + 自研后端：前端工程质量高，Vue3 / TypeScript / monorepo / 多 UI 版本成熟；后端可用 Spring Boot 或 NestJS 自己控制业务边界。
- SoybeanAdmin + 自研后端：Vue3 / Vite / TypeScript / Naive UI 体验好，适合做更现代的管理后台；后端需要自行实现权限和业务 API。
- vue-pure-admin + 自研后端：Vue3 / Vite / Element Plus / TypeScript，提供精简版，适合项目型后台前端；后端仍需自研。
- COOL Admin：提供 Node / Java / Vue 管理框架，模块化、插件化、CRUD 快速开发，适合希望保留 Node.js 路线时验证。
- JHipster：生成式全栈框架，Spring Boot + Vue/React/Angular 等选择多，适合强工程规范项目，但对本项目的家庭 NAS 场景可能偏重。

初步倾向：

- 若目标是最快得到可用后台，优先验证 RuoYi-Vue SpringBoot3 + Vue3 TypeScript。
- 若目标是管理端体验和前端工程质量更强，优先验证 Vben Admin 或 SoybeanAdmin + 自研 Spring Boot / NestJS 后端。
- 若目标是保持 Node.js 全栈，验证 COOL Admin Node 或 NestJS + Vben Admin。
- 不优先选择微服务版、低代码平台、工作流平台或多租户重型框架，除非后续业务明确需要。

必须验证后再定：

- 许可证是否允许当前用途。
- 本地 Windows / Docker 启动是否顺畅。
- 登录、菜单、权限、接口鉴权是否容易保留。
- 新增照片目录、播放策略、设备管理模块是否容易接入。
- 后台任务、文件扫描、缩略图生成、AI 任务、WebSocket 是否容易扩展。
- 是否容易把 `tv-player-web` 作为独立前端接入同一个后端 API。

#### 已定技术路线

- 管理后台采用 Vben Admin 的 Ant Design Vue 应用作为后台底座。
- 后端采用 NestJS，自研照片、播放、设备、AI 和任务 API。
- 前后端共享 TypeScript 类型，优先在 `packages/shared` 中维护播放单元、策略、设备和 AI 结果类型。
- 电视端采用 Android TV APK 作为真实安装入口和最终展示端。
- Android TV 主播放器采用 Kotlin + Jetpack Compose for TV 原生实现，不以 WebView 作为正式播放核心。
- 当前 `apps/tv-player-web` 调整为后台播放预览、浏览器调试和备用 WebView 模式，不再作为电视端主播放器。
- Android TV 端参考 Immich Android TV 和 Aerial Views 的照片播放、屏保、D-pad、Overlay 和兼容性经验，但不直接 fork 作为项目底座。

#### 管理后台

- 基于 `apps/web-antd` 改造，保留 Vben 的布局、主题、权限、动态菜单、请求封装和工程规范。
- 删除或隐藏与本项目无关的示例模块。
- 新增本项目业务菜单：照片目录、照片库、AI 文案、播放策略、设备管理、播放预览、日志状态、系统设置。
- 管理后台优先服务配置和排障，不做复杂相册交互。

#### Android TV 电视端

MVP 推荐：

- Kotlin + Jetpack Compose for TV。
- AndroidX TV Material。
- Coil 图片加载。
- Retrofit / OkHttp 网络访问。
- DataStore 保存后台地址和设备 Token。
- Compose Animation + graphicsLayer + Canvas 实现照片转场和文案动画。
- Room + 本地文件缓存用于离线播放兜底。
- 电视端启动后支持后台地址配置、设备绑定码、原生播放页、离线提示和性能降级。

长期正式版：

- Android TV 原生照片播放器。
- 双缓冲图片层 + Overlay 文案层。
- 动画模板、版式模板、屏保模式、防烧屏 Overlay 漂移。
- 低端设备可降级为淡入淡出，关闭复杂动画。

说明：照片播放不建议使用 ExoPlayer 作为主核心。ExoPlayer 适合视频，不适合图片转场、文字 Overlay 和版式动画。照片播放核心应使用图片渲染层 + 动画系统 + 文字图层。

Android TV 详细技术选型见 `往日重现_AndroidTV技术选型.md`。

## 4. 云端 AI API 选型

### 4.1 推荐优先级

#### 第一选择：OpenAI Vision

适合生成自然、克制、有情绪的家庭回忆文案。建议用于：

- 图片场景理解
- 人物数量描述，不做人脸身份识别
- 标题生成
- 一句话生成
- 情绪标签
- 场景标签
- 推荐版式
- 推荐动画

#### 第二选择：Google Gemini Vision

适合大批量图像理解和成本控制场景。可作为备用 Provider。

#### 第三选择：Anthropic Claude Vision

适合需要更长文本分析、细腻描述、文案审核时使用。

#### 补充选择：Azure AI Vision

适合结构化视觉能力，例如 OCR、对象标签、地标、基础图像分析。若只是家庭回忆文案，不建议作为唯一文案生成模型，需要结合 LLM。

### 4.2 AI Provider 抽象

后端必须设计统一 AI Provider 接口，避免绑定单一厂商。

```ts
interface VisionProvider {
  analyzePhoto(input: AnalyzePhotoInput): Promise<AnalyzePhotoResult>;
}

interface AnalyzePhotoInput {
  photoId: string;
  imagePath: string;
  promptStyle: 'warm_memory' | 'poetic' | 'family_diary' | 'travel' | 'minimal';
  locale: 'zh-CN' | 'en-US';
}

interface AnalyzePhotoResult {
  title: string;
  caption: string;
  sceneTags: string[];
  moodTags: string[];
  objects: string[];
  peopleCount?: number;
  locationHint?: string;
  suggestedLayout: string;
  suggestedImageTransition: string;
  suggestedTextAnimation: string;
  confidence?: number;
  raw?: unknown;
}
```

### 4.3 AI 输出要求

AI 必须返回结构化 JSON，不允许只返回自然语言。

标准输出：

```json
{
  "title": "那年的夏天",
  "caption": "有些快乐不用解释，照片已经替我们记住了。",
  "sceneTags": ["家庭", "旅行", "户外"],
  "moodTags": ["温暖", "轻松"],
  "objects": ["天空", "道路", "人物"],
  "peopleCount": 3,
  "locationHint": "海边或湖边",
  "suggestedLayout": "bottom_gradient",
  "suggestedImageTransition": "ken_burns_fade",
  "suggestedTextAnimation": "fade_up",
  "confidence": 0.82
}
```

### 4.4 AI 使用策略

- 默认批量预生成，不实时生成。
- 每张照片仅首次生成一次。
- 用户修改过的文案必须锁定，不可被自动覆盖。
- 支持重新生成。
- 支持多风格文案版本。
- 生成失败要记录日志。
- API Key 只能保存在后台，不下发到电视端。
- 每日调用上限可配置。
- 支持并发数限制。
- 支持失败重试。

## 5. 核心业务流程

```text
启动 Docker Compose
↓
访问后台地址
↓
初始化管理员账号
```

```text
进入照片目录管理
↓
添加目录

↓
测试连接
↓
保存目录
↓
开始扫描
↓
生成照片索引和缩略图
```

### 5.3 照片扫描流程

```text
读取目录文件
↓
过滤支持格式
↓
生成 Photo ID
↓
读取文件基础信息
↓
读取 EXIF
↓
生成缩略图
↓
保存 photo 记录
↓
标记待 AI 生成
```

### 5.4 AI 文案生成流程

```text
选择待生成照片
↓
读取缩略图或压缩图
↓
调用云端 Vision API
↓
返回结构化 JSON
↓
校验 JSON
↓
保存到 ai_caption 表
↓

```

```text

↓
电视端显示绑定码
↓
后台设备管理输入绑定码
↓
命名设备
↓
分配播放策略
↓
电视端收到绑定成功事件
↓
进入播放页
```

### 5.6 播放策略配置流程

```text
创建播放策略
↓
选择照片目录
↓
配置筛选条件
↓
配置播放模式
↓
配置播放速度
↓
配置版式和动画
↓
绑定设备
↓
保存并推送
```

### 5.7 电视端播放流程

```text
读取设备 Token
↓
获取设备策略
↓
获取播放列表
↓
预加载当前照片和下一张
↓
渲染图片层
↓
渲染渐变遮罩层
↓
渲染 AI 文案层
↓
执行图片转场动画
↓
执行文字动画
↓
播放结束后上报记录
↓
切换下一张
```

## 8. 播放引擎设计

### 8.1 Android TV 原生图层结构

```text
Root Player
├── Background Blur Layer，可选
├── Previous Image Layer
├── Current Image Layer
├── Next Image Preload Layer
├── Gradient Mask Layer
├── Caption Layer
├── Metadata Layer
├── Control Overlay Layer
└── Error / Loading Layer
```

### 8.2 关键机制

1. 当前图片和下一张图片必须双缓冲。
2. 下一张图片必须提前预加载。
3. 图片加载失败自动跳过并上报错误。
4. 文字动画与图片转场分离。
5. 动画模板由后台下发。
6. 低端设备可降级为淡入淡出。
7. 长时间播放必须释放上一张图片资源。
8. 网络失败时优先使用缓存播放列表。

### 8.3 默认动画模板

```json
{

  "textEnterDurationMs": 900,
  "textIdle": "soft_float",
```

## 11. 开发计划

新版详细开发计划见 `往日重现_详细开发计划.md`。本基础文档只保留阶段总览，具体开发任务、验收标准、风险和每次开发后的更新要求以详细计划文档为准。

### 阶段 0：工程底座与播放链路基线

核心任务：完成 Vben Admin、NestJS、Web 播放页、共享类型和最小播放 API，验证“后台 API -> 播放列表 -> 播放页 -> 播放记录上报”链路。

当前状态：已基本完成。

### 阶段 1：Docker 后台端可部署闭环

核心任务：补全 `backend-api`、`admin-web`、数据库、Redis、媒体缓存和 NAS 挂载目录的 Docker 生产部署结构。

### 阶段 2：Android TV APK MVP

核心任务：新增 `apps/android-tv`，采用 Kotlin + Compose for TV 实现后台地址配置、设备绑定、原生照片播放器、遥控器按键适配和基础异常状态。

### 阶段 3：真实数据模型与数据库迁移

核心任务：建立照片目录、照片、AI 文案、设备、播放策略、播放记录和任务日志等核心模型，并用数据库替换内存样例数据。

### 阶段 4：NAS 照片扫描、缩略图和 EXIF

核心任务：读取真实 NAS / 本地挂载照片目录，完成扫描、索引、去重、EXIF、缩略图和媒体读取接口。

### 阶段 5：管理后台业务模块

核心任务：将 Vben Admin 改造成项目后台，实现照片目录、照片库、设备管理、播放策略、播放预览、日志和系统设置。

### 阶段 6：AI 文案生成与播放策略完善

核心任务：实现 AI Provider、批量生成、文案编辑锁定、策略筛选、动画版式和设备策略刷新。

### 阶段 7：稳定性、打包发布和长期运维

核心任务：优化 Docker 镜像、Android TV APK 打包签名、离线缓存、性能降级、长时间播放测试、备份恢复和部署说明。

## 12. Codex 开发注意事项

1. 不要把 AI 文案直接合成进图片。
2. 不要让电视端直接调用云端 AI API。
3. 不要只返回图片 URL，播放接口必须返回完整播放单元。
4. 不要删除 NAS 原图，所有操作只影响索引和播放状态。
5. 不要在电视端做复杂管理。
6. 不要用 ExoPlayer 作为照片播放主引擎。
7. 不要把 WebView 作为正式 Android TV 主播放器，只能作为后台预览、调试和备用模式。
8. 不要让 Android TV 直接加载 NAS 原图，必须优先使用后端生成的电视展示图。
9. 不要把本地播放设置直接覆盖后台策略，必须遵守策略优先级。
10. 所有耗时任务必须进入队列或后台任务。
11. 所有 AI 结果必须可编辑、可锁定、可重新生成。
12. 所有接口必须考虑设备离线和照片读取失败。
